## Flymap Architecture & Coding Rules

### Purpose
A concise, repeatable blueprint to build new Flutter apps with the same architecture, style, and tooling as Flymap.

## Project Structure
- lib/
  - entity/          Domain models (Equatable-only data)
  - data/
    - api/          Remote APIs + mappers
    - local/        Local DB services + mappers
    - route/        Deterministic computations (e.g., great-circle)
    - tiles_downloader/ Vector tiles download/MBTiles logic
    - network/      Cross-cutting network utilities (e.g., ConnectivityChecker)
  - repository/      Bridges use cases to data layer
  - usecase/         Application/business logic (single responsibility)
  - ui/
    - screens/       Feature-first folders (screen + viewmodel + widgets)
    - map/           Map utilities/layers/components
    - theme/         App-wide theme, colors, text styles
  - router/          go_router configuration
  - di/              GetIt registrations
  - logger.dart      Lightweight logging facade
  - firebase_options.dart Generated by FlutterFire
- assets/            Styles, sprites, glyphs, data, icons
- test/              Unit and widget tests

## State Management
- Use Cubit from flutter_bloc for each feature (e.g., FlightPreviewCubit).
- Define sealed state classes per feature in a dedicated state file.
- All state and entity classes must extend Equatable and include all relevant fields in props.
- Emit explicit, meaningful states; never overload a single state for multiple phases.

## Dependency Injection
- Use GetIt for DI.
- Register:
  - Lazy singletons for shared services (e.g., DB, repositories, connectivity, use cases).
  - Factories for lightweight mappers and providers.
- Centralize registrations in di/di_module.dart; no ad-hoc GetIt.register in feature code.

## Routing
- Use go_router with a single `AppRouter.createRouter()` that declares all routes.
- Use named routes for analytics and clarity.
- Wire `FirebaseAnalyticsObserver(FirebaseAnalytics.instance)` into router observers.

## Networking & Backend
- Cloud Functions: wrap calls in data/api/ with isolated mappers.
- Map API calls to domain models via explicit mapper classes; never pass raw maps around.
- Surface errors to the Cubit; log details with `Logger` and show user-friendly messages in UI.

## Local Data
- Sembast for app data (e.g., flights). Use mappers to serialize/deserialize:
  - Include createdAt/updatedAt ISO timestamps.
  - Store nested entities as maps/lists, not strings.
- Sqflite only for MBTiles (tile storage). Create MBTiles schema explicitly.

## Vector Tiles & MBTiles
- Downloader uses isolates. Communicate via ReceivePort/SendPort with typed DTOs.
- Insert tiles sequentially to avoid DB write conflicts.
- Progress events every N tiles; a final verifying phase runs after workers finish.
- Enforce success threshold:
  - If tilesDownloaded/totalTiles < 0.70, delete the `.mbtiles`, emit an error, and stop.
- Verify MBTiles > 0 bytes before success; prefer additional integrity checks where feasible.

## Connectivity
- Provide a `ConnectivityChecker` service.
- Gate network-heavy flows (e.g., downloads) both at start and mid-flow as needed.
- On offline, emit a domain-appropriate error state with a clear message (e.g., “No internet connection…”).

## Analytics
- Initialize Firebase with `DefaultFirebaseOptions.currentPlatform` in `main.dart`.
- Use `FirebaseAnalyticsObserver` for route screen views.
- Keep analytics opt-out and DebugView instructions handy for QA; do not collect ad IDs.

## UI Guidelines
- Feature-first structure: screen folder contains viewmodel (Cubit), state, widgets (small components), and any feature map widgets if needed.
- Extract reusable pieces to widgets/; avoid monolithic screen files.
- Always guard async callbacks and external controller usage with `mounted` checks and null checks.
- MapLibre:
  - Use `onStyleLoaded` to add layers.
  - Dispose controllers in `dispose()` and guard against post-dispose usage.
- Error UX:
  - Display inline error banners/messages during long-running operations (e.g., download).
  - Provide explicit actions: Retry, Cancel, Back.

## Theming & Assets
- Centralize colors, text, and theme in ui/theme/.
- Reference sprites/glyphs/styles via file:// paths when offline; ensure they’re copied to cache before map init.
- Keep assets declared in pubspec.yaml; maintain clear folder semantics (styles/, sprites/, glyphs/, data/).

## Logging & Errors
- Use `Logger('ComponentName')` consistently; no print in production paths.
- Log at boundaries (start, success, failure) with useful context (IDs, counts).
- Never swallow errors silently; convert to user-visible state with actionable messaging.

## Use Cases
- Single responsibility per file (e.g., GetFlightInfoUseCase, DownloadMapUseCase).
- Do not perform UI manipulation; return events or values to Cubits.
- Handle happy path and error path explicitly; avoid broad catches without messaging.

## Repositories
- Keep repositories thin: orchestrate calls to services and aggregate results.
- Avoid business logic here; push logic to use cases.

## Entities & Mappers
- Entities are immutable, Equatable, and domain-focused.
- Mappers are the only place that knows serialization details (DB/JSON).
- When adding fields (e.g., createdAt), update: entity, mapper.toDb, mapper.fromDb, and usage points.

## Testing
- Unit test mappers and use cases.
- Widget test Cubit-to-UI rendering states and error flows.
- Prefer deterministic tests (no network/isolates) by mocking services.

## Build & Tooling
- Lints: use flutter_lints; keep analyzer clean for new changes.
- Prefer `flutter analyze` locally and in CI.
- For non-interactive logs (e.g., CI), piping to `| cat` disables fancy TTY spinners without changing behavior.

## Code Style
- Naming:
  - Classes: Nouns (FlightPreviewCubit), functions: verbs (startDownload), variables: descriptive nouns.
  - Avoid short/cryptic names; prefer clarity over brevity.
- Types:
  - Strong typing; no `dynamic` unless absolutely required at serialization boundaries.
- Control flow:
  - Early returns; shallow nesting; explicit error handling.
- Comments:
  - Explain “why”, not “how”. No TODOs—implement or ticket.
- Formatting:
  - Match project formatting; multi-line over complex one-liners.

## Patterns to Repeat
- Cubit + sealed states per feature.
- Use case orchestrates work; repository delegates to services; mappers serialize.
- DI via GetIt; never instantiate services in feature code.
- Connectivity guarded flows; inline user feedback for errors.
- Success thresholds and verification for long-running or batch processes.

## Patterns to Avoid
- Business logic in Widgets or repositories.
- Passing raw maps/JSON beyond mappers.
- Global state without DI; service locators inside feature code.
- Silent failures; hidden background tasks without state updates.

## Template Snippets

### Add a new use case
```dart
class DoThingUseCase {
  final SomeRepository _repo;
  DoThingUseCase({required SomeRepository repo}) : _repo = repo;

  Future<ResultType> call(Params params) async {
    // Validate inputs
    // Call repo/services
    // Map/return domain entity
  }
}
```

### Cubit skeleton
```dart
class FeatureCubit extends Cubit<FeatureState> {
  final DoThingUseCase _doThing;
  FeatureCubit({required DoThingUseCase doThing})
    : _doThing = doThing, super(const FeatureLoading());

  Future<void> load() async {
    emit(const FeatureLoading());
    try {
      final data = await _doThing.call(...);
      emit(FeatureLoaded(data));
    } catch (e) {
      emit(FeatureError('Failed to load: $e'));
    }
  }
}
```

### GetIt registration
```dart
void register() {
  i.registerLazySingleton<ConnectivityChecker>(() => const ConnectivityChecker());
  i.registerFactory<SomeMapper>(() => SomeMapper());
  i.registerLazySingleton<SomeService>(() => SomeService(...));
  i.registerLazySingleton<SomeRepository>(() => SomeRepository(service: i.get()));
  i.registerLazySingleton<DoThingUseCase>(() => DoThingUseCase(repo: i.get()));
}
```

### Route setup with analytics
```dart
GoRouter(
  observers: [FirebaseAnalyticsObserver(analytics: FirebaseAnalytics.instance)],
  routes: [/* ... */],
);
```
